1. 엔티티는 인스턴스의 집합이다. (테이블은 속성의 집합이다.)
관계는 페어링의 집합이다. 
=> 엔티티와 관계를 인스턴스와 페어링이라고 부르기도 한다.

엔티티 - 릴레이션 - 테이블/ 인스턴스 - 튜플 - 행(Row)/ 속성 - 속성 - 행(Column)


2. pk를 기본식별자라고 부른다. pk가 하나라면 단일 식별자, 두 개 이상이라면 복합 식별자라고 부른다.


3. 도메인은 속성값의 범위이다. 데이터 타입이나 제약조건을 의미한다.


4. 관계는 엔티티간의 연관이다. 부모 엔티티와 자식 엔티티가 1:N 관계라면 자식 엔티티는 부모 엔티티로부터 기본 식별자를 내려받는다. 이를 외래 식별자라고 한다.(fk)


5. 카디널리티는 하나의 부모 엔티티가 몇 개의 자식 인스턴스(행)와 페어링될 수 있는지를 나타낸다. 
=> 1:1/ 1:n/ n:n,  만약 n:n관계라면 하나의 엔티티를 주어서 1:n관계를 만들어주는 것이 좋다.


6. 옵셔널리티는 부모 인스턴스와 자식 인스턴스의 페어링 여부를 나타낸다. (부모 행과 자식 행의 연결 상태)
페어링이 되어야 하면 필수관계, 페어링이되지 않아도 되면 선택 관계라고 부른다.


7. 관계유형
관계유형은 식별 관계와 비식별 관계로 구분된다. a엔티티와 b엔티티가 존재한다고 가정하자. a엔티티의 기본 식별자(pk)를 b엔티티의 기본 식별자로 받으면 식별 관계라고 한다. 반면 a엔티티의 기본 식별자를 b엔티티가 일반 속성으로 받으면 비식별 관계라고 한다.


8. 테이블 선
1. 점선이면 비식별관계/ 실선이면 식별관계이다.
2. 카디널리티에 의해 1:n이 구성되었다면 1은 일직선 다쪽은 까마귀 발이다.
3. 옵셔널리티에 의해 선택 관계가 구성되었다면 (부서는 사원을 가지지 않아도 되지만 사원은 부서에 속해야 할 경우)
사원 테이블 선에 동그라미 표시를 한다.


9. 정규형
데이터 이상 현상을 제거하기 위한 관계형 모델의 설계 지침이다.

1정규형 => 한 칼럼이 다중 값을 가지는 경우 ex) 상품 테이블이 존재하고 칼럼이 관련부서목록일 경우 관련부서목록은 번호,사원,부서이름 같은 값이 존재할 수 있다. 데이터 타입이 다를 경우를 생각해도 좋다.

2정규형 => 일반 속성이 식별자의 일부 속성에만 종속되는 경우 ex) 주문상세 테이블이 존재하고 pk로 주문번호와 상품코드가 존재한다. 일반속성으로 주문수량과 상품명이 존재한다. 이때 주문수량은 pk2개에 종속되지만 상품명은 상품코드 속성에만 종속된다면 2정규형 위반이라고 부른다. 즉, pk에 종속되지 않은 칼럼이 존재하면 2정규형 위배이다.

3정규형 => 이행종속된 경우(일반 속성이 다른 일반 속성에 종속되는 경우) 일반 칼럼이 pk처럼 행동하고 일반 칼럼이 이 pk처럼 행동하는 것에 종속된 경우를 말한다.


10. 반정규화
성능 개선을 위해 의도적으로 데이터를 중복시키는 것을 말한다. 테이블의 pk를 다른 테이블의 일반 속성으로 내려주는 행위
